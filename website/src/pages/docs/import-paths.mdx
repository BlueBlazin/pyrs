<DocSection>
	This page describes how PYRS resolves stdlib roots and import paths in CLI and script execution.
</DocSection>

<DocSection title="Path Resolution Rules">
	<DataTable headers={["Rule", "Behavior"]}>
		<tr>
			<td><code>PYRS_CPYTHON_LIB</code></td>
			<td>
				When set, PYRS treats it as the explicit stdlib root and resolves CPython stdlib modules from
				that location.
			</td>
		</tr>
		<tr>
			<td>Host <code>lib-dynload</code> fallback</td>
			<td>
				Used only when the configured stdlib root does not contain its own dynamic-loader directory.
			</td>
		</tr>
		<tr>
			<td>Script directory</td>
			<td>
				In script mode, the script’s directory participates in import resolution similar to CPython.
			</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Common Local Workflow">
	<CommandBlock
		commands={[
			"export PYRS_CPYTHON_LIB=.local/Python-3.14.3/Lib",
			"pyrs -c \"import sys, dataclasses; print(sys.version.split()[0])\"",
		]}
	/>
</DocSection>

<DocSection title="Troubleshooting Imports">
	<ul class="docs-list">
		<li>If a stdlib module is missing, verify `PYRS_CPYTHON_LIB` points to a valid CPython 3.14 `Lib` tree.</li>
		<li>For environment-specific issues, test with `pyrs -S -c ...` to isolate startup import effects.</li>
		<li>Keep virtualenv activation separate from parity harness runs if you need stable baseline behavior.</li>
	</ul>
	<Callout title="Note">
		PYRS targets CPython semantics, so import behavior differences should be treated as parity bugs and
		reported with a minimal reproducer.
	</Callout>
</DocSection>
