<DocSection>
	The browser playground lives at <a href={withBasePath("/playground/")}><code>/playground</code></a>.
	It auto-loads the website's <code>wasm-vm-probe</code> bundle, pre-registers a curated stdlib subset
	from <code>/wasm/stdlib_subset_v1.json</code>, and provides a minimal REPL-focused interface. Runtime
	execution is handled in a dedicated Web Worker so command execution does not run on the page UI thread.
</DocSection>

<DocSection title="Quick Start">
	<CodeBlock
		label="Open and run"
		code={`# 1) Build browser wasm assets (repo root)
./scripts/build_wasm_website_bundle.sh

# 2) Start website dev server
pnpm --dir website dev

# 3) Open /playground, wait for runtime loaded, and run input`}
	/>
	<Callout title="Input model" tone="note">
		Press <code>Enter</code> to execute. Use <code>Shift</code> + <code>Enter</code> for multiline input.
	</Callout>
	<Callout title="Scope">
		The playground is for fast exploration, not a full native install. It ships a curated stdlib subset
		and reports unsupported modules/capabilities explicitly instead of silently falling back.
	</Callout>
</DocSection>

<DocSection title="What The Playground Loads">
	<DataTable headers={["Component", "Behavior"]}>
		<tr>
			<td>Wasm runtime</td>
			<td>The website bundle is built with <code>wasm-vm-probe</code>, so runtime execution is enabled in the worker.</td>
		</tr>
		<tr>
			<td>Stdlib content</td>
			<td>A curated virtual stdlib subset is registered before the REPL session starts; full CPython stdlib is not shipped in-browser.</td>
		</tr>
		<tr>
			<td>Unsupported surfaces</td>
			<td>Snippets that need missing modules or unsupported runtime capabilities return explicit stderr/blocker output.</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Runtime Status Signals">
	<DataTable headers={[
		"State",
		"Meaning",
		"What to do",
	]}>
		<tr>
			<td><code>loading</code></td>
			<td>The wasm bundle is still being initialized.</td>
			<td>Wait for ready state before running snippets.</td>
		</tr>
		<tr>
			<td><code>ready</code></td>
			<td>Runtime loaded, worker ready, and the curated stdlib subset has been registered.</td>
			<td>Run snippets normally.</td>
		</tr>
		<tr>
			<td><code>error</code></td>
			<td>The runtime failed to initialize.</td>
			<td>Check browser console and confirm wasm assets are published.</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Output Shape">
	<DataTable headers={[
		"Output class",
		"Description",
	]}>
		<tr>
			<td>Input transcript</td>
			<td>Every submitted command is echoed with <code>&gt;&gt;&gt;</code> / <code>...</code> prompts.</td>
		</tr>
		<tr>
			<td>stdout</td>
			<td>Normal execution output from the wasm runtime.</td>
		</tr>
		<tr>
			<td>stderr</td>
			<td>Runtime errors, compile/syntax failures, missing-module messages, or unsupported capability messages.</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Debug Route">
	For low-level diagnostics and worker/state contract probing, use the unlisted

</DocSection>
