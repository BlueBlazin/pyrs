<DocSection>
	The browser playground lives at <a href={withBasePath("/playground/")}><code>/playground</code></a>.
	It auto-loads the website's <code>wasm-vm-probe</code> bundle, pre-registers a curated stdlib subset
	from <code>/wasm/stdlib_subset_v1.json</code>, and provides a minimal REPL-focused interface. Runtime
	execution is handled in a dedicated Web Worker so command execution does not run on the page UI thread.
</DocSection>

<DocSection title="Quick Start">
	<CodeBlock
		label="Open and run"
		code={`# 1) Build browser wasm assets (repo root)
./scripts/build_wasm_website_bundle.sh

# 2) Start website dev server
pnpm --dir website dev

# 3) Open /playground, wait for runtime loaded, and run input`}
	/>
	<Callout title="Input model" tone="note">
		Press <code>Enter</code> to execute. Use <code>Shift</code> + <code>Enter</code> for multiline input.
	</Callout>
	<Callout title="Scope">
		The playground is for fast exploration, not a full native install. It ships a curated stdlib subset
		and reports unsupported modules/capabilities explicitly instead of silently falling back.
	</Callout>
</DocSection>

<DocSection title="What The Playground Loads">
	<DataTable headers={["Component", "Behavior"]}>
		<tr>
			<td>Wasm runtime</td>
			<td>The website bundle is built with <code>wasm-vm-probe</code>, so runtime execution is enabled in the worker.</td>
		</tr>
		<tr>
			<td>Stdlib content</td>
			<td>A curated virtual stdlib subset is registered before the REPL session starts; full CPython stdlib is not shipped in-browser.</td>
		</tr>
		<tr>
			<td>Unsupported surfaces</td>
			<td>Snippets that need missing modules or unsupported runtime capabilities return explicit stderr/blocker output.</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Supported Modules">
	<Callout title="Two sources of support" tone="note">
		The older 26-module wasm stdlib plan is only the seed list. The current playground supports modules from
		two sources: a curated CPython source pack published with the website, and native/bootstrap modules
		compiled into the Rust VM. For example, <code>urllib.parse</code> comes from the curated pack, while
		<code>itertools</code> is available from the VM bootstrap without adding extra stdlib pack bytes.
	</Callout>
	<Callout title="Public modules only" tone="note">
		The lists below intentionally omit private and internal helper modules such as <code>_collections_abc</code>
		and <code>re._casefix</code>. Those helpers may still exist behind the scenes, but this page only lists the
		public-facing import surface that users should rely on.
	</Callout>
	<DataTable headers={["Bucket", "Current scope", "Notes"]}>
		<tr>
			<td>Curated stdlib source pack</td>
			<td>53 public modules</td>
			<td>
				Loaded from <a href={withBasePath("/wasm/stdlib_subset_manifest_v1.json")}><code>/wasm/stdlib_subset_manifest_v1.json</code></a>
				and pre-registered before the REPL becomes ready.
			</td>
		</tr>
		<tr>
			<td>Native/bootstrap modules</td>
			<td>30 additional public modules</td>
			<td>
				Exposed by the VM bootstrap/runtime layer in <code>src/vm/vm_bootstrap_import.rs</code>. These are
				importable in browser mode even when they are not part of the curated source pack.
			</td>
		</tr>
		<tr>
			<td>Capability-blocked modules</td>
			<td>7 public modules</td>
			<td>
				These are rejected by the browser capability policy instead of partially working.
			</td>
		</tr>
	</DataTable>
	<Callout title="Importability vs capability parity" tone="note">
		If a module is listed below, it is currently part of the wasm REPL import surface. That does not mean
		every API inside it has native parity: browser mode still blocks sockets, subprocesses, dynamic library
		loading, and terminal-only behavior at call time.
	</Callout>
	<CodeBlock

	/>
	<CodeBlock
		label="Additional native/bootstrap modules available in wasm today (30 public modules)"
		code={`__future__, array, asyncio, atexit, binascii, collections.abc, errno
faulthandler, gc, itertools, marshal, math, mmap, os, os.path, pathlib, platform

unicodedata, uuid, zlib`}
	/>
	<DataTable headers={["Blocked public modules", "Blocker key", "Why blocked in browser mode"]}>
		<tr>
			<td><code>ctypes</code></td>
			<td><code>dynamic_library_load</code></td>
			<td>Dynamic library loading is not available in the wasm host.</td>
		</tr>
		<tr>
			<td><code>numpy</code>, <code>scipy</code></td>
			<td><code>dynamic_library_load</code></td>
			<td>Standard distributions depend on native extension binaries.</td>
		</tr>
		<tr>
			<td><code>socket</code></td>
			<td><code>network_sockets</code></td>
			<td>Raw socket capability is unavailable in browser mode.</td>
		</tr>
		<tr>
			<td><code>subprocess</code>, <code>multiprocessing</code></td>
			<td><code>process_spawn</code></td>
			<td>Browser mode does not expose process creation or subprocess spawning.</td>
		</tr>
		<tr>
			<td><code>readline</code></td>
			<td><code>interactive_terminal</code></td>
			<td>The browser playground does not provide a TTY-style interactive terminal surface.</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Runtime Status Signals">
	<DataTable headers={[
		"State",
		"Meaning",
		"What to do",
	]}>
		<tr>
			<td><code>loading</code></td>
			<td>The wasm bundle is still being initialized.</td>
			<td>Wait for ready state before running snippets.</td>
		</tr>
		<tr>
			<td><code>ready</code></td>
			<td>Runtime loaded, worker ready, and the curated stdlib subset has been registered.</td>
			<td>Run snippets normally.</td>
		</tr>
		<tr>
			<td><code>error</code></td>
			<td>The runtime failed to initialize.</td>
			<td>Check browser console and confirm wasm assets are published.</td>
		</tr>
	</DataTable>
</DocSection>

	<DataTable headers={[
