	This page focuses on how to use each path effectively, and what behavior to expect when debugging.

<DocSection title="Prompt and Multiline Input Model">
	<p>
		Use the primary prompt (`>>>`) for new statements. The continuation prompt (`...`) indicates
		you are inside an unfinished block or multiline construct.
	</p>
	<DataTable headers={["Prompt", "Meaning"]}>
		<tr>
			<td><code>&gt;&gt;&gt;</code></td>
			<td>Primary prompt for a new statement.</td>
		</tr>
		<tr>
			<td><code>...</code></td>
			<td>Continuation prompt while completing a block or multi-line construct.</td>
		</tr>
	</DataTable>
	<CodeBlock
		label="REPL Session"
		code={`>>> from dataclasses import dataclass
>>> @dataclass
... class User:
...     name: str
...     id: int
...
>>> User("ada", 7)
User(name='ada', id=7)`}
	/>
</DocSection>

<DocSection title="Execution State and Session Behavior">
	<ul class="m-0 list-disc space-y-1.5 pl-5">
		<li>Imports and variable assignments persist for the lifetime of the REPL session.</li>
		<li>Each entered statement executes in the current session state.</li>
		<li>Restart REPL to clear state when debugging contamination between experiments.</li>
	</ul>
	<CodeBlock
		label="State Persistence Example"
		code={`>>> counter = 0
>>> counter += 1
>>> counter
1
>>> counter += 1
>>> counter
2`}
	/>
</DocSection>

<DocSection title="Error Feedback in Interactive Runs">
	<p>
		Syntax and runtime failures are rendered with traceback context in CPython-style shapes. For
		repeatable bug reports, capture the exact failing snippet and include both PYRS and CPython output.
	</p>
	<CodeBlock
		label="Runtime Error Shape"
		code={`>>> items = {}
>>> items["missing"] + 1
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
KeyError: 'missing'`}
	/>
</DocSection>

<DocSection title="Exit, Interrupt, and Deterministic Runs">
	<ul class="m-0 list-disc space-y-1.5 pl-5">
		<li><code>Ctrl+D</code> exits the REPL on macOS/Linux terminals.</li>
		<li><code>Ctrl+C</code> interrupts the running statement where supported.</li>
		<li>For deterministic automation, prefer `-c` or script-file execution over interactive REPL entry.</li>
		<li>Use script mode for reproducible benchmarks and CI runs.</li>
	</ul>
	<Callout title="Practical Workflow">
		Develop quickly in REPL, then move stable snippets into script files once behavior is confirmed.
		This keeps iteration fast without sacrificing reproducibility.
	</Callout>
</DocSection>

<DocSection title="Troubleshooting Checklist">
	<ul class="m-0 list-disc space-y-1.5 pl-5">
		<li>If interactive behavior differs from script mode, rerun with `pyrs -c ...` to isolate session state effects.</li>
		<li>If imports fail unexpectedly, verify your import root and `PYRS_CPYTHON_LIB` setup.</li>
		<li>When reporting REPL bugs, include prompt transcript and exact terminal platform details.</li>
	</ul>
