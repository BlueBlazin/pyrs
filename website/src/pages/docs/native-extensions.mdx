<DocSection>
	Native extension support is an active workstream focused on C-API closure and scientific-stack
	bring-up. PYRS targets a disciplined, CPython-semantics-first extension substrate rather than ad-hoc
	compatibility shims.
</DocSection>

<DocSection title="Compatibility Modes">
	<DataTable headers={["Mode", "Status"]}>
		<tr>
			<td><code>pyrs314</code> (source-build mode)</td>
			<td>Primary supported mode for extension bring-up.</td>
		</tr>
		<tr>
			<td><code>cp314</code> wheel-compat reuse</td>
			<td>Deferred until ABI/runtime conformance evidence is sufficient.</td>
		</tr>
	</DataTable>
</DocSection>

<DocSection title="Current Implemented Surface">
	<DataTable headers={["Surface", "Current state"]}>
		<tr>
			<td>Manifest-backed extension loading (<code>.pyrs-ext</code>)</td>
			<td>Implemented and smoke-covered.</td>
		</tr>
		<tr>
			<td>Direct shared-object import (<code>.so</code>/<code>.dylib</code>/<code>.pyd</code>)</td>
			<td>Implemented with <code>pyrs_extension_init_v1</code> and <code>PyInit_*</code> fallback.</td>
		</tr>
		<tr>
			<td>CPython single-phase init compatibility slice</td>
			<td>Partial baseline implemented; full C-API breadth remains open.</td>
		</tr>
		<tr>
			<td>PEP 489 multi-phase init</td>
			<td>Not closed yet.</td>
		</tr>
	</DataTable>
</DocSection>
